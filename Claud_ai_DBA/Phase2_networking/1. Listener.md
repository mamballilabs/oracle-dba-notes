what is a service 
is  a name the client uses to connect to specific database or PDB
```
Client says:    "I want to connect to service ORCLCDB"
Listener says:  "ORCLCDB is handled by instance ORCLCDB, status READY"
Listener:       routes connection to that instance
```

how did services get registered 
Remember LREG from your background processes output?
```
[oracle@ol9ai ~]$ lsnrctl status showed 121 processes
One of them was:
LREG    →   Listener Registration process
```
LREG is the background process that **automatically registers** your instance and all its services with the listener at startup. You do not have to do anything manually — LREG handles it.
```
Instance starts
      │
      ▼
LREG wakes up
      │
      ▼
LREG contacts listener on port 1521
      │
      ▼
Registers:
  - Instance name ORCLCDB
  - Service ORCLCDB
  - Service orclpdb1
  - Service ORCLCDBXDB
      │
      ▼
Listener now knows about all services
      │
      ▼
Clients can connect
```

question 2:You saw `sqlplus / as sysdba` works without a listener. Your junior DBA says the listener is down and nobody can connect. Is he completely right?

```
Junior DBA says "nobody can connect"

Actually:
Local connections    →   still work via IPC   ✅
Remote connections   →   completely broken    ❌
Application servers  →   cannot connect       ❌
Other DBAs remotely  →   cannot connect       ❌
```

Listener down = remote connections fail. Local OS connections still work.
This is an important distinction in production incidents. When users call saying they cannot connect — first check if the listener is running before assuming the database is down.

file 3: sqlnet.ora : the network behavior rules 

```bash
NAMES.DIRECTORY_PATH= (TNSNAMES, ONAMES, HOSTNAME)
```

This one line controls how Oracle resolves connection names. It defines the order Oracle tries when you type a connect string:

Step 1 → TNSNAMES look in tnsnames.ora first 
Step 2 → ONAMES try Oracle Names server (old, rarely used) 
Step 3 → HOSTNAME try resolving as a plain hostname

So when you type `sqlplus system/password@ORCLCDB
Oracle checks tnsnames.ora → ORCLCDB found → use it → connect

If not in tnsnames.ora
Oracle tries ONAMES → not configured → skip
Oracle tries HOSTNAME → treats ORCLCDB as a hostname → fails

Your sqlnet.ora is minimal — just one line. Production sqlnet.ora files are much richer:
```
Encryption settings 
Authentication methods (Kerberos, RADIUS) 
Timeout values 
Firewall keepalive settings 
SSL/TLS certificates
```
