every time a session connects to oracle , oracle needs a server process to serve that session

Does each session get its own dedicated process — or do many sessions share a pool of processes?

option 1: dedicated server

we saw this in tnsnames.ora
```
(SERVER = DEDICATED)
```

how it works 
```
Session 1 connects  →  Oracle spawns Server Process 1
Session 2 connects  →  Oracle spawns Server Process 2
Session 3 connects  →  Oracle spawns Server Process 3

Session 1 disconnects  →  Server Process 1 destroyed
Session 2 disconnects  →  Server Process 2 destroyed
```

one session one server process always together 
```
Memory cost:
Each server process uses PGA memory
100 sessions = 100 server processes = 100 PGAs
```

option 2: shared server

many session share a small pool of server processes

```
100 sessions connect
        │
        ▼
Only 10 server processes created
        │
        ▼
Sessions take turns using server processes
When session is idle  →  process serves another session
When session needs work  →  picks up next available process

Memory cost:
10 server processes = 10 PGAs
100 sessions share them = much less memory
```

the trade off

```
DEDICATED SERVER          SHARED SERVER
─────────────────         ──────────────────
One process per session   Many sessions per process
Higher memory usage       Lower memory usage
Best for active sessions  Best for mostly idle sessions
Simpler architecture      More complex
Your current setup        Needs dispatcher config
```

when to use each 

```
Use DEDICATED when:
├── Sessions are actively working most of the time
├── OLTP applications — constant queries
├── DBA connections — always
└── Most production databases today

Use SHARED when:
├── Thousands of mostly idle connections
├── Legacy thin client applications
├── Connection count exceeds OS process limits
└── Rare in modern environments
```

Modern applications use **connection pooling** at the application layer — WebLogic, JDBC pool, application server pools. The pool maintains a small number of active database connections and shares them across thousands of application users.

```
10,000 application users
        │
        ▼
Application connection pool
(WebLogic, JDBC, etc.)
        │
        ▼
50 dedicated Oracle connections
        │
        ▼
50 dedicated server processes
50 PGAs
```

The application layer solves the many-users problem before it reaches Oracle. So Oracle dedicated server handles a manageable number of connections efficiently.

```sql
SELECT server,

```

```sql
SELECT name, value
FROM   v$parameter
WHERE  name IN (
       'shared_servers',
       'max_shared_servers',
       'dispatchers')
ORDER BY name;

NAME                      VALUE
------------------------- --------------------------------------------------
dispatchers               (PROTOCOL=TCP) (SERVICE=ORCLCDBXDB)
max_shared_servers
shared_servers            1
```

Three things to read here:

### shared_servers = 1

Oracle has started **one** shared server process. But nobody is using it — all your sessions are dedicated. This one shared server exists specifically for the XDB service.

### max_shared_servers = empty

No upper limit set. If shared server was heavily used Oracle could grow the pool automatically.

### dispatchers = ORCLCDBXDB

This is the XML DB dispatcher — a special shared server setup Oracle creates automatically for its internal XML DB HTTP service. Nothing you configured. Nothing you need to touch.

```
Your database is running in DEDICATED mode   ✅
One shared server exists for XDB only        ✅ (Oracle internal)
No application sessions using shared server  ✅
```

question
1. "The listener is down. Which users are affected and which are not?"
"Remote users and applications connecting via TCP are completely cut off. Local connections using IPC — including `sqlplus / as sysdba` — continue working. This is why DBAs can always reach the database locally during a listener outage."

2. "What is the difference between tnsping succeeding and a successful sqlplus connection?"
```
tnsping confirms:
├── TNS alias resolves        ✅
├── Listener is running       ✅
└── Service is registered     ✅

tnsping does NOT confirm:
├── Database is open          ❌
├── Credentials are valid     ❌
└── User has connect privilege❌

Classic trap — tnsping says OK, but sqlplus fails. Always means authentication or database state issue, not a network issue.
```

"tnsping only confirms the network path is clear — TNS resolves, listener responds, service is registered. A successful sqlplus connection additionally requires valid credentials, the database to be open, and the user to have connect privilege. tnsping passing but sqlplus failing always points to authentication or database state — not networking."

3. "A developer says — our application has 10,000 users but we only want 50 database connections. How does Oracle handle this?"

application-side connection pooling

```
Shared server — Oracle manages the pool
     │
     ├── Added complexity inside Oracle
     ├── Not suitable for all application types
     ├── Rare in modern production
     └── Oracle themselves recommend against it
         for most workloads

Application connection pool — managed outside Oracle
     │
     ├── WebLogic connection pool
     ├── JDBC connection pool
     ├── Application server manages 50 connections
     ├── 10,000 app users share those 50 connections
     └── Oracle sees only 50 dedicated connections
         clean, simple, industry standard
```

"Modern solution is application-side connection pooling — WebLogic, JDBC, or application server pools maintain 50 dedicated Oracle connections and share them across 10,000 application users. Oracle sees 50 dedicated server processes with 50 PGAs. Shared server is an older Oracle-side solution rarely used today because application pooling handles this more cleanly outside the database."
